# Visual Tuning, Presets, And Playback UI Acceptance Criteria

This document captures the accepted behavior for the current visual tuning and playback UI branch. It is additive to `../Usage ACs.md`.

## VT-1 Visual Tuning Panel

- **VT-1.1 Default visibility:** The visual tuning panel is closed by default.
- **VT-1.2 Toggle:** The `Tuning` control opens and closes the panel without resetting current tuning values.
- **VT-1.3 Dragging:** The panel can be repositioned by dragging its header, and normal slider/select interaction inside the panel must not start a drag.
- **VT-1.4 Responsive layout:** The panel uses responsive columns and avoids an awkward fixed internal scroll on common desktop widths.
- **VT-1.5 Extendable controls:** Adding a new numeric visual tuning parameter must require updating the typed default config and the control metadata, not duplicating slider UI code.
- **VT-1.6 Export tuning:** The `Copy config` control copies the current `State.visualTuning` values to the clipboard as valid preset JSON under a `visualTuning` property, including a version field.
- **VT-1.7 Copy fallback and feedback:** If the Clipboard API is unavailable or rejects the write, the UI uses a DOM textarea fallback copy path. Successful copy attempts show a short, non-layout-shifting `Copied` status.

## VT-2 Tuning Parameters

- **VT-2.1 Defaults:** Existing visual defaults remain the baseline output.
- **VT-2.2 Expanded ranges:** All exposed panel parameters have materially larger but bounded ranges than the previous implementation.
- **VT-2.3 Large circle strokes:** Circle shockwave and ring stroke controls allow very large stroke values.
- **VT-2.4 Circle background:** Circle background hue and opacity can be adjusted independently from core circle hue and opacity.
- **VT-2.5 Global background:** Background red, green, and blue controls can produce black, the existing dark background, and pure white.
- **VT-2.6 Music sensitivity:** A single `Music sensitivity` slider scales the already-established audio and visual feature values up or down.
- **VT-2.7 Analyzer isolation:** Music sensitivity must not re-run or mutate offline audio analysis results.
- **VT-2.8 Modulation range:** The modulation bus must clamp `kineticTension`, `densityDrive`, `spectralChaos`, `rhythmicImpulse`, and `macroMomentum` to `0.0..1.0`.
- **VT-2.9 Sensitivity scaling:** `audioSensitivity` scales modulation bus outputs linearly until a value reaches the normalized upper bound.
- **VT-2.10 Morphing target:** Sliders and preset selection write to `State.targetTuning`; the renderer interpolates `State.visualTuning` toward that target during draw.
- **VT-2.11 Wormhole group:** A `Wormhole` control group exposes `wormholeRadius`, `wormholeDepth`, `wormholeSpeed`, `wormholeWarp`, `wormholeCurve`, `wormholeRing`, `wormholeDepthCoherence`, `wormholeContinuity`, `wormholeStarfield`, and `wormholeGalaxy`. `wormholeCurve` can fully zero curvature; `wormholeDepthCoherence` deterministically compresses immutable depth cohorts without accumulated morph/seek damage; `wormholeContinuity` is independent of ring alignment; and the two background masters remain global because bundled presets do not write them. New parameters follow VT-1.5 (typed default config plus control metadata, no duplicated slider UI).
- **VT-2.11 Hero lane placement:** The tuning panel exposes `heroLaneBottomOffset`, which controls the vertical placement of the Hero visual identity's horizontal event lane as a normalized distance from the bottom of the viewport.
- **VT-2.12 Hero metronome tuning:** The tuning panel exposes `heroBeepVolume`, `heroBeepMode`, and `heroEventMode`. `heroBeepMode` uses discrete values `0=Off`, `1=Quarter`, `2=Off-beat`, `3=Triplets`, and `4=Syncopated`. `heroEventMode` uses discrete values for all audio events, audio drums only, or metronome beeps only.
- **VT-2.13 Hero discrete morphing:** `heroBeepMode` and `heroEventMode` snap instantly during preset morphing and must not be interpolated through floating-point intermediate values. `heroBeepVolume` remains a continuous interpolated tuning value.
- **VT-2.14 Wormhole emission morphing:** `wormholeEmissionMode` remains a discrete UI selector (`0..2`) but is a continuous tuning value during morphing. Intermediate values must not snap; `wormholeEmissionGain()` evaluates the adjacent integer modes and crossfades their gains deterministically.

## VT-3 Presets

- **VT-3.1 Preset folder:** Presets are stored as JSON files under `public/visual-tuning-presets/`.
- **VT-3.2 Manifest:** The app reads the preset list from `public/visual-tuning-presets/index.json`.
- **VT-3.3 File names as names:** Preset names are derived from JSON file names.
- **VT-3.4 Clean labels:** The preset selector does not show the `.json` suffix.
- **VT-3.5 Default selection:** The selector starts from `default.json` when it is present instead of showing a placeholder.
- **VT-3.6 Load on select:** Selecting a different preset immediately applies that preset's parameters.
- **VT-3.7 Backward compatibility:** Missing preset fields are filled from `defaultVisualTuning` during load.
- **VT-3.8 Stable state reference:** Loading a preset mutates the existing shared tuning object instead of replacing it with a new reference.
- **VT-3.9 Quiet success:** Successful preset loads do not display transient success text that shifts the UI.
- **VT-3.10 Error visibility:** Failed manifest or preset loads may show an error state long enough to diagnose the problem.
- **VT-3.11 Transient status scope:** Transient status text is allowed for copy feedback and preset-load failures when it is placed in an existing fixed-size or non-shifting status area and clears itself after a short timeout.
- **VT-3.12 Visual mode compatibility:** Presets may include `visualMode`. Values matching a supported visual identity route through `requestVisualModeChange()`, update logical `State.visualMode` synchronously, and update the `#visual-mode` select. Missing or unsupported values must not fail the preset load.
- **VT-3.13 Semantic automation preset mapping:** Performance-plan generation must prefer metadata scoring from preloaded preset JSON over hard-coded preset filenames when `GeneratorOptions.presetMetadata` is available. Custom preset names are valid automation targets when their tuning or dramaturgy parameters match the target section profile.
- **VT-3.14 Automation-triggered morph authority:** When performance automation loads a preset, the active projected point's `intensity`, scaled `morphDurationSec`, and `morphCurve` override morph values supplied by the preset's `visualTuning` or `morphProfile`. Manual preset selection does not provide this override and remains unchanged. An asynchronous automation preset response must be discarded when its point id is no longer active.

## VT-4 Header And Metrics

- **VT-4.1 Track metadata:** The header shows only the audio file title after load.
- **VT-4.2 No analysis chatter:** The header does not show completion text, sample count, section count, cue count, or similar analysis internals.
- **VT-4.3 BPM location:** BPM appears in the `#bpm-header-badge` next to the loaded track title, not as a metric card.
- **VT-4.4 Metrics toggle:** A compact metrics toggle is placed above the seekbar, not in the top-right control group.
- **VT-4.5 Metrics layout:** Metrics use the original card-like grid layout and adapt by viewport width.
- **VT-4.5a Metrics default state:** The metrics grid is hidden by default on initial load. The metrics toggle starts with `aria-expanded="false"` and reveals the grid only after user interaction.
- **VT-4.6 Cue removal:** The cue metrics block is not displayed.
- **VT-4.7 Dynamics state size:** `Dynamics State` occupies one metrics grid unit.
- **VT-4.8 Frame projection labels:** The visible `Density` and `Melody Presence` metric labels project the current `AudioFrame.densityProj` and `AudioFrame.melodyProj` values. `AudioFrame.fxProj` remains the canonical fx-presence projection for modulation, but the duplicate fx-presence dashboard entry has been removed in favor of the canonical `FX` feature card.
- **VT-4.9 Removed metric cards:** BPM, Progress, and FX Presence are not present in the metrics grid. BPM is handled by the header badge, Progress by the seekbar/time display, and FX Presence by the canonical `FX` card.

## VT-5 Seekbar

- **VT-5.1 Full width:** The seekbar spans the available window width responsively.
- **VT-5.2 Visual style:** The seekbar uses the same minimal rectangular visual language as metric value cards.
- **VT-5.3 Interaction:** Dragging or clicking the seekbar updates the visible scrub position immediately, but audio seeking is committed through the audio engine only when the interaction ends.
- **VT-5.4 Same-file reload:** The file input is cleared immediately after its `change` event reads the selected file, so choosing the same audio or video file again triggers a new load. This must remain a `PlaybackController` DOM-input behavior and must not move playback ownership out of `AudioEngine`.

## VT-6 Playback Surface

- **VT-6.1 Center play:** A responsive translucent play affordance appears in the center when playback is available and stopped or paused.
- **VT-6.2 No pause overlay:** A separate pause overlay is not shown during playback.
- **VT-6.3 Double-click toggle:** Double-clicking the visual background toggles play and pause.
- **VT-6.4 Single-click behavior:** Single click on the visual background does not pause playback.
- **VT-6.5 Keyboard play/pause:** Pressing `Space` toggles play and pause when the visual surface is the active element.
- **VT-6.6 Keyboard seek:** `ArrowLeft` and `ArrowRight` seek backward and forward by five seconds when the visual surface is the active element.
- **VT-6.7 Focus scope:** Playback keyboard shortcuts must not hijack typing or control interaction outside the visual surface.
- **VT-6.8 Fullscreen toggle:** The fullscreen control toggles the document between normal and fullscreen presentation using the browser Fullscreen API, including vendor-prefixed fallbacks where needed.
- **VT-6.9 Reactive video rate:** With a loaded video backplate, normal playback may modulate muted video playback rate from `State.modulation.macroMomentum` and `State.modulation.rhythmicImpulse`, clamped to `0.5x..2.0x`. Export, pause, stop, and clear paths must keep or restore deterministic `1.0x` behavior.
- **VT-6.10 Video color sampling:** The dashboard video path must sample the current video frame through a low-resolution offscreen canvas and write averaged RGB into `State.videoDominantColor` without full-resolution frame reads.

## VT-7 Looping

- **VT-7.1 Default loop:** Playback is in loop mode by default.
- **VT-7.2 Toggle:** The top control can switch between looped and one-shot playback.
- **VT-7.3 Natural end loop:** In loop mode, natural track end restarts playback from the beginning.
- **VT-7.4 Natural end once:** In one-shot mode, natural track end stops playback and emits the normal ended state.

## VT-8 Auto-Hide Chrome

- **VT-8.1 Idle hide:** Top controls, bottom playback chrome, and visible panels fade out after inactivity.
- **VT-8.1a Initial lock:** The dashboard starts in locked-visible chrome state. Chrome auto-hide does not run until the user intentionally unpins the UI with the visual-surface single-click gesture.
- **VT-8.2 Reveal:** Mouse, pointer, keyboard, or touch activity reveals the chrome.
- **VT-8.3 Hover persistence:** Hovering or focusing chrome keeps it visible even after the idle timer elapses. If the timer fires while chrome is hovered, hiding is rescheduled instead of applying `body.chrome-idle`.
- **VT-8.4 State preservation:** Collapsed panels stay collapsed after reveal; open panels return only if they were open.
- **VT-8.5 Cursor hide:** The mouse cursor is hidden while the chrome is in the idle-hidden state.
- **VT-8.6 UI pinning:** A single click on the visual background, after the double-click detection window expires, toggles a pinned chrome state that keeps the UI visible and disables idle auto-hide.
- **VT-8.7 UI unpinning:** A subsequent single click on the visual background restores the normal idle auto-hide behavior.
- **VT-8.8 Fast unpin feedback:** When that subsequent single click unpins the chrome, the hide timer uses a fast `400ms` delay. Standard inactivity and hover/focus recovery use the shorter standard delay of about `1400ms`; hover-triggered deferral reschedules with the fast `400ms` delay. This is UI chrome behavior, not renderer, analyzer, or playback behavior.

## VT-9 Regression Requirements

- **VT-9.1 File loading:** Audio file loading, analysis, play enablement, and seek enablement must continue to work after UI changes.
- **VT-9.2 Preset switching:** Switching presets after audio load must not reset playback state or require reloading the track.
- **VT-9.3 Contract tests:** The shared tuning and worker contracts must remain covered by automated tests.
- **VT-9.4 Build:** The TypeScript build must pass after changes to state, UI, audio, or visual contracts.
- **VT-9.5 Interactive dramaturgy timeline:** The seekbar chrome includes a compact canvas timeline that visualizes precomputed `TrackAnalysis` data without runtime audio analysis. `drawTimelineGridlines` must draw BPM-derived bar lines; sections, RMS, buildup, spectral pivot regions, tension trends, significant cues, and the playhead must be layered clearly and remain readable in compact, resized, and overlay modes.
- **VT-9.6 Decode failure UI:** If browser-level audio decoding or file loading fails before worker analysis can complete, the UI must show a file-load error, keep playback and seek controls disabled, and re-enable file selection.
- **VT-9.7 Resizable timeline panel:** The timeline can be resized from its top handle. The resize path clamps height to safe bounds, redraws through the throttled timeline draw path, preserves HDPI canvas sharpness, and remembers the last expanded height for returning from overlay mode.
- **VT-9.8 Fullscreen overlay structure:** Opening the timeline overlay applies `.timeline-overlay-active` to the full `.seek-container`, `.is-fullscreen-overlay` to `.timeline-wrapper`, and `body.timeline-overlay-open` to the document body. The overlay fills the viewport above other UI, hides unrelated chrome, and closes back to the prior seekbar position and height.
- **VT-9.9 HTML tooltip interaction:** Hovering the timeline displays the `#timeline-tooltip` HTML element near the pointer. The tooltip reports the current time and zoom, matching section, bar index and state, RMS, BarAnalysis bass/mid/treble spectral-band ratios, buildup pressure, tension trend, and nearby cue data where available. Analyzer confidence, alternate tempo details, section/cue `reasons`, and the novelty debug overlay may appear only behind the `analyzerDebugOverlay` feature flag.
- **VT-9.10 DAW-style zoom and pan:** Mouse-wheel interaction zooms the timeline viewport around the pointer. The max zoom is dynamic and shared by `DashboardUI` and `TimelineCanvas`: `max(16, duration / 5.0)`, preserving at least about five visible seconds while keeping the previous `16x` floor. Normal left click or drag always scrubs/seeks the playhead, including when zoomed. Shift-drag or middle-button drag pans the visible viewport. During playback, a zoomed viewport follows the playhead when it leaves the `15%..75%` visible range. `State.zoom` and `State.pan` remain the canonical timeline viewport state.
- **VT-9.11 Scrub buffering performance line:** Timeline and seekbar dragging must not call `AudioEngine.seek()` repeatedly. UI drag updates `scrubTime`, the visible time, the seekbar value, and a yellow scrub playhead through the throttled draw path. A single final audio seek is committed on `pointerup`, `pointercancel`, `change`, or equivalent touch-end interaction.
- **VT-9.12 Renderer hot-path optimization:** The p5 render loop must not run O(N) `findIndex` searches over beat events or cue arrays while paused, stopped, or at natural track end. Visual event indexes are synchronized only through the event-driven `syncEventIndex` callback registered with `addPositionChangedListener`.
- **VT-9.13 Performance Automation Plan display:** The timeline renders automation zones from the scaled plan view derived from `State.editedPerformancePlan` (falling back to `State.performancePlan` when no edits exist). Partially visible automation zones must not disappear at timeline viewport edges. Each `PerformanceAutomationPoint` zone shows a draggable vertical start handle, a morph-end resize handle, a horizontal sensitivity handle, and a shaded morph curve area for whole-zone sliding. Projected geometry is display/hit-test state only; every mutation targets the base point by id. `intensity` maps to the local sensitivity override (`0.1..4.0`). Morph curve math uses unclipped x coordinates and canvas clipping; the segment count is at least `15` and grows with the visible curve width. Preset-derived colors mark the automation zone, morph curve, intensity line, and sensitivity handle state, with all automation fills capped at `0.08` alpha. Selection must use outline plus restrained glow rather than a darker fill; the zone segment after the morph remains dimmed. RMS/bar lines must remain connected at viewport edges. Runtime scheduling applies the projected point's sensitivity and morph authority through the preset target. Optional analyzer scheduling metadata (`analysisConfidence`, `timingMode`) must not break rendering or plan validation; `timingMode` accepts `bar-aligned`, `novelty`, and `energy-reactive`. `State.sectionOverrides` is fully removed.
- **VT-9.16 Waveform zoom projection:** Timeline waveform peaks are precomputed into a `Float32Array` waveform cache by `TimelineCanvas.setAudioBuffer()`. Bucket resolution targets `80 Hz` and is capped at `500000` buckets. Deep zoom sampling uses linear interpolation between buckets so the waveform stays smooth at high zoom. This remains a cached projection and must not introduce runtime audio analysis.
- **VT-9.14 Spectral Pivot and Drop Anticipation overlays:** The timeline draws `spectralPivot` active regions with a magenta dotted overlay and draws the configured `dropAnticipation` look-ahead window as a magenta suspense gradient.
- **VT-9.15 Visual identity deterministic smoke:** Built-in visual identities must be covered by browser-free mock-render tests over multiple genre profiles. The test must assert no crashes and deterministic backend draw-call counts for repeated runs.
- **VT-9.17 Dramaturgy copy/load:** `Copy Dramaturgy` and `Load Dramaturgy` controls on the dramaturgy timeline must transfer the current performance-automation plan through the clipboard using the DOM-free `src/automation/dramaturgyTransfer.ts` module. Copy serializes `State.editedPerformancePlan` (falling back to `State.performancePlan`) into a tagged JSON envelope, using the same Clipboard API write with DOM-textarea fallback as `Copy config`. Load must accept a tagged envelope, a bare plan, or a `Copy config` payload that embeds a `performancePlan`; validate and normalize it (sort points by time, strip unknown fields, clamp confidence, and preserve+normalize the optional renderer-independent Visual OS `PerformanceAutomationPoint.meta` provenance without ever rejecting a point because of it); report a precise error in the timeline status line for empty, non-JSON, non-plan, wrong-version, or malformed-point input; and only apply the plan after an overwrite confirmation, after which it preloads referenced presets and redraws the timeline. The clipboard read must degrade to a manual-paste fallback when the Clipboard API is unavailable. The serialization, validation, and edge cases must be covered by `tests/dramaturgy-transfer.test.mjs`.
- **VT-9.19 Visual OS generator controls (ADR-005):** When the `Dramaturgy` generator strategy is selected, the timeline header shows the Visual OS controls: a style-pack selector populated from `public/visual-tuning-presets/style-packs.json`, a substyle selector for the resolved pack, and an Activity selector (`Macro` / `Balanced` / `Active`, default `Balanced`). The selectors are hidden for the `Strict Alternating` and `Hero Rhythm` strategies. Changing the top-right Visual Mode drives the default style pack (1:1 id mapping except `temporal` -> `base-temporal`); the pack selector stays an explicit override the Visual Mode re-aligns. A missing or unparseable `style-packs.json` leaves the pack list empty and generation falls back to the legacy generator without throwing. The Activity level is forwarded to the generator and deterministically controls SceneEvolution waypoint density; it is a normal user-facing control, not a feature flag.
- **VT-9.20 Micro-choreography planner and automation envelopes (ADR-005 extension):** A `Variation` selector (`Stable` / `Paired` / `Expressive`, default `Paired`) sits beside Activity and is forwarded to the adapter as `DramaturgyVariantMode`. Variation controls choreography COMPLEXITY/style (behaviour-vocabulary size, cycle frequencies, lifetime, seeded jitter); Activity (`Macro`/`Balanced`/`Active`) is the orthogonal DENSITY control with two levers — a density scale on the planner's target segment length (`Active` shortens segments = denser, `Macro` lengthens = sparser) AND a hard cap on segments per scene (1/5/8). `Active` therefore genuinely fills a scene with more points, not merely permitting a higher ceiling; `Macro` also suppresses release points. For each scene the adapter classifies a deterministic, renderer-independent `AutomationSituation` from already-generated narrative/behaviour/duration (never raw audio: `automationSituationClassifier`), resolves a behaviour vocabulary of OPAQUE handles (authored `behaviourVocabulary[situation]` → the situation's `variantPairs` → the scene's narrative handle, all inheritance-resolved like `targetMap`), and the `microChoreographyPlanner` returns an ordered set of `ChoreographySegment`s. Each segment carries a behaviour role (`primary`/`secondary`/`release`/`sparse`/`focus`), an OPAQUE target, a relative intensity, and an `AutomationEnvelope` (`attack`/`sustain`/`release`/`cooldown` summing to the segment duration). Subdivision is adaptive (situation, bar structure, variation, length, timing confidence) and snaps interior boundaries to bars under a reliable grid (equal-time fallback otherwise); the role sequence is a seeded forward cycle with weighted recency memory, not a fixed A/B. The concrete preset binding stays adapter-only: handles resolve via `targetMap`, degrading to the pack default for a missing/typo handle without throwing.
- **VT-9.21 Automation breathing (envelope lifetime):** Automation lifetime is bounded by its segment and never auto-stretches to the next point. `PerformanceAutomationPoint.morphDurationSec` comes from the segment envelope's `attackSec`; `finalize` only shortens a morph on overlap (anti-overlap), leaving the trailing `cooldown` as visible "air" so a sparse scene's zones do not run edge-to-edge. A separate softer release point is emitted only for a `release`-role segment with a long-enough decay, and only when Activity is not `Macro`. This holds in every mode, including `Stable`.
- **VT-9.22 Stable is not static / Variation vs Activity:** `Stable` always produces a plan (never a single frozen anchor): a long scene yields multiple points sharing ONE visual identity (one resolved preset) with varying intensity/envelope across the scene (a breathing macro-arc). `Paired` and `Expressive` introduce additional behaviour families (`Expressive` reaches more distinct families than `Paired`); Activity alone controls the total point density (`Macro < Balanced < Active`), where `Active` measurably increases the point count on a medium (20-30s) drop/break rather than just raising the cap.
- **VT-9.23 Deterministic choreography provenance:** Generation is fully deterministic - identical `TrackAnalysis` + settings produce a byte-identical plan - using a seed hashed from the analysis (`hash(trackSeed, sceneIndex, segmentIndex)`), with no runtime randomness. Each `PerformanceAutomationPoint.meta` carries renderer-independent provenance (`automationSituation`, `vocabularyId`, `variantRole`, `movementGesture`, `longScenePhase`, `globalArcRole`, and the opaque `targetStateReference`), surfaced in the timeline debug tooltip and round-tripped by Copy/Load. Invalid meta enum values are dropped without rejecting an otherwise valid point. The planner names no preset filename or tuning key. Breathing, adaptive bar-snapped subdivision, cycle grammar, memory, determinism, Stable-edge variation, and Variation-vs-Activity separation are covered by `tests/visual-os.test.mjs` and `tests/dramaturgy-transfer.test.mjs`.
- **VT-9.24 Movement grammar and style movement vocabulary:** Every non-empty `ChoreographySegment` carries one abstract `MovementGesture` from the governed union (`pulse`, `drive`, `orbit`, `scatter`, `collapse`, `expand`, `bloom`, `fragment`, `ripple`, `slice`, `tunnel`, `swarm`, `lock`, `echo`, `fade`). The pure `movementGrammar` resolves the gesture from situation, role, behaviour, narrative, variation mode, previous gesture, and use counts. A resolved style/substyle `movementVocabulary[situation]` constrains the candidate set; inheritance is per situation, substyles may override, and a list containing only unknown gestures must retain the valid inherited/default fallback. No movement-domain module may name a preset, target handle, tuning key, p5, DOM, renderer API, or runtime state.
- **VT-9.25 Plan-local cross-scene variation memory:** Each `adaptScenePlanToPerformancePlan()` call creates a fresh `VariationMemory`; no memory survives across plan builds. It records recent targets, gestures, situations, family/gesture counts, the latest peak gesture, and latest drop vocabulary, and exposes defensive snapshots to planners. When alternatives exist, adjacent scene boundaries avoid the same target/gesture and overused families receive deterministic penalties. Returning drops evolve rather than replaying an identical target/gesture sequence; outro resolution may recall a previous peak through an `echo`/`fade`-biased interpretation. Mutating a snapshot must not mutate the generator memory.
- **VT-9.26 Long-scene internal form:** `longScenePlanner` returns one `entry` section for scenes shorter than 24 seconds. A long energetic scene is divided into `entry`, `establish`, `intensify`, `peak`, and `release`; a long reflective scene uses `entry`, `develop`, `counter`, and `decay`. Section durations cover the scene exactly and carry preferred roles, preferred style-permitted gestures, and intensity bias. Micro-choreography maps segments to these phases while preserving the Activity point cap, so a long drop gains a readable internal arc instead of only gaining more points.
- **VT-9.27 Global visual narrative:** `globalVisualNarrative` consumes `VisualScenePlan` only and must never inspect raw audio. It deterministically emits a track arc type, total energy shape, primary/secondary gesture families, return strategy, optional climax index, and one `SceneNarrativeBias` per scene (`setup`, `development`, `climax`, `aftermath`, `resolution`). Multiple drops receive different return bias, the highest-energy scene is marked as climax, and an outro is marked as resolution. The result feeds situation classification, long-scene planning, movement grammar, and micro-choreography without changing the adapter-only preset binding or the legacy fallback.
- **VT-9.28 Adaptive transition dynamics:** The Visual OS adapter computes a pure `TransitionDynamicsProfile` from already-produced `TrackAnalysis`. Low-contrast smooth evidence lengthens attack; transient/high-contrast evidence shortens attack. Adaptation preserves envelope total, release, and cooldown and changes generated morph duration through `attackSec`. Explicit scene/target curves remain authoritative, while implicit defaults may adapt to `easeInOut` or `exponential`. Missing evidence is a deterministic no-op. Coverage lives in `tests/transition-dynamics.test.mjs` and the Visual OS integration assertions.
- **VT-9.29 Morph Scale projection and editor boundary:** The timeline-level Morph Scale slider writes `State.automationMorphScale`. `computeMaxMorphScale`, `clampMorphScale`, and `applyMorphScale` create a safe non-destructive plan projection used by `TimelineCanvas` and runtime scheduling. Hit-testing may read projected geometry, but editor selection, preset/curve/intensity edits, time drag, zone drag, morph resize, and delete resolve the base point by `id` through `automationPlanEditing.ts`. Inspector duration is projected for display and divided by the active scale before base-plan storage.
- **VT-9.30 Automation and wormhole regression coverage:** `tests/morph-scale.test.mjs` covers safe scaling, determinism, cache field signatures, point ordering, insertion, and deletion. `tests/automation-plan-editing.test.mjs` covers scaled selection followed by preset/duration edit, drag, delete, invalidation, and projection reload persistence. `tests/wormhole-lifecycle.test.mjs` covers fractional emission modes and morph-duration-aware automation response; `tests/morphing.test.mjs` covers non-snapping emission-mode interpolation. Timeline/UI/contracts and semantic integration tests cover projection wiring and automation morph authority.
- **VT-9.18 Visual Score DSL:** With `semanticResolver` enabled, accepted track analysis must produce a deterministic, JSON-serializable `VisualScorePlan` with motif memory, confidence-aware subdivisions, and one typed transition per adjacent motif. The derived choreography carries optional motif and transition fields while preserving the existing action API. With the flag disabled, legacy performance-plan automation remains the only owner of preset-driven `targetTuning` changes.

## VT-10 Render And Stream Output

- **VT-10.1 Render backend boundary:** Effect modules must draw through `VisualRendererBackend`; direct p5 drawing calls belong in `P5RendererBackend` or p5-owned primitives.
- **VT-10.2 Chroma key modes:** `chromaKeyMode` supports normal background, green chroma background, and transparent clearing for overlay capture.
- **VT-10.3 Low-latency mode:** `performanceMode` disables expensive glow paths such as radial gradient drawing.
- **VT-10.4 Presentation URL:** Loading the app with `?presentation=true` hides UI chrome by setting the shared UI visibility state to false.
- **VT-10.5 Timeline inspection modes:** The dramaturgy timeline supports compact, manually resized, and fullscreen overlay inspection modes. The existing expanded CSS height remains a fallback sizing state, but the top-right timeline control opens and closes the fullscreen overlay instead of performing audio or playback work.
- **VT-10.6 Visual identity registry:** `PlexusRenderer` must delegate to renderer-owned `IdentityTransitionController`, which resolves styles through `StyleRegistry`; no hard-coded mode drawing branches are allowed. Steady state draws only the selected identity directly, while an active transition draws the recorded outgoing/incoming pair.
- **VT-10.6a Identity replacement:** During playback/export, a valid mode change must use two persistent renderer-owned targets and a song/export-time smoothstep crossfade `A * (1 - alpha) + B * alpha`. Additive blending is forbidden. Paused/stopped mode changes switch without dual rendering.
- **VT-10.6b Ownership and simulation:** `requestVisualModeChange()` is the only runtime writer of `State.visualMode` and owns transition record creation/replacement. `IdentityTransitionController` owns transition consumption/completion and is the only compositor caller. Exactly one eligible identity may advance shared particles/shockwaves per transition frame; effects cannot own targets, compositing, or transition state.
- **VT-10.7 Built-in visual modes:** The UI must expose `classic`, `temporal`, `dark-techno`, `organic-ambient`, `cyberpunk`, and `cosmic-wormhole` in `#visual-mode`, plus `hero` when `featureFlags.heroEffect` is enabled.
